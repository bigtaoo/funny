# client/ 关键约束与核心模块

## 关键约束

- **渲染**：`pixi.js-legacy`，兼容微信小游戏 WebGL
- **游戏逻辑**：纯 TS，固定点数（`game/math/fixed.ts`），与渲染解耦
- **平台适配**：Web / 微信小游戏 / CrazyGames，多入口 webpack 构建
- **骨骼动画 Runtime**：`StickmanRuntime`（`src/render/stickman/`），加载 `.tao` ZIP，驱动 PIXI Sprite
- **确定性约束**：游戏逻辑（`client/src/game/`）内严禁 `Math.random()`，必须用 `Prng`（`game/math/prng.ts`）
- **多语言（i18n）**：`zh.ts` 为键唯一来源（`TranslationKey`），`en`/`de` 声明为 `Record<TranslationKey, string>` 漏翻报错；`t(key, params?)` 取词，支持 `{param}` 插值
- **网络协议 codegen**：`transport.proto`/`game.proto` → ts-proto via buf → `src/net/proto/`；`openapi.yml` → openapi-typescript → `src/net/openapi.ts`；改契约须重跑 `npm run proto:gen` / `npm run rest:gen`
- **统一输入管线（M13）**：`InputSource` 接口，`LocalInputSource`（单机）/ `NetInputSource`（联机锁步）/ `RecordingInputSource`（录制）/ `ReplayInputSource`（回放）
- **⚠️ 渲染层销毁契约（防内存泄漏）**：每个战斗视图（`GameRenderer` 及子视图 `BoardView`/`UnitView`/`BuildingView`/`HandView`）**必须**实现 `destroy()`：①注销所有挂在 `PIXI.Ticker.shared` 上的一次性特效 tick（`Ticker.shared` 是 GC 根，漏一个就钉住整局场景图+纹理）；②`pool.drain()` 销毁池内已 `removeFromParent` 的游离对象；③`container.destroy({children:true})`。**共享纹理（spritesheet/`bake()`/`Texture.from`）只解引用不销毁。** 详见 **[`client-memory-leak.md`](client-memory-leak.md)**（事故复盘 + 完整契约 + heap snapshot 验证法）—— 2026-06 一次因六个视图全无 `destroy()` 导致每局退场泄漏整张场景图、2 小时涨到 16GB 的事故。
- **⚠️ 高频 `render()` 场景的文本纹理销毁（2026-06-27 事故）**：画布场景用「`container.removeChildren()` + 全量重画」模式重渲。`removeChildren()` **不销毁**子对象——而 `sketchUi.txt()` 每次 `new PIXI.Text`，**每个 Text 自带一张 GPU 纹理**，孤儿纹理要等 PIXI 纹理 GC（默认 ~60s）才回收。`LoginScene` 尤其致命：**每次按键** + 光标闪烁 **每 0.5s** 都全量 render，连打字时纹理瞬间堆满 → iPad Safari WebGL 内存预算极小 → 上下文丢失 → Safari 刷新页面（用户反馈「注册输昵称一直崩溃」）。**修复契约**：用 `sketchUi.tearDownChildren(container)`（共享单一实现）代替裸 `removeChildren()`——遍历返回的子对象，`PIXI.Text` 走 `destroy({texture:true,baseTexture:true})` 释放自有纹理，其余（Graphics/包 `bake()` 共享纹理的 Sprite）走 `destroy({children:true})`（缺省 `texture:false`，**不碰共享烘焙纹理**）。
  - **关键依赖（已被测试钉死）**：`destroy({children:true})` 会递归销毁子树，故**嵌套**的 Text（如聊天气泡 `container→layer→node→body`）也会被命中——`PIXI.Text.destroy()` 会与它自带的 `defaultDestroyOptions(texture:true,baseTexture:true)` 合并，即便经 `{children:true}` 递归也会释放自有纹理；而嵌套的非 Text Sprite 因 `texture` 缺省 false 保住共享纹理。PIXI 升级若改动该默认值会悄悄复发泄漏。
  - **审计结论（2026-06-27）**：命中并已套用契约 = `LoginScene`（逐键+光标 0.5s）/ `ChatScene`（逐键+光标 0.5s，含嵌套气泡）/ `ShopScene`（充值码逐键+光标 0.5s+按钮补间逐帧）/ `SettingsScene`（改名逐键+光标 0.5s）/ `SectScene`·`FamilyScene`（创建表单逐键→`bodyLayer`）/ `WorldMapScene`（HUD 每 ~5s 行军轮询 + 行军面板每 1s 倒计时）。**豁免** = `AuctionScene`：input 仅存字符串不触发 render，`update()` 仅 toast 计时，render 全是事件驱动（切页/滚动/拉取），无逐键/逐帧/定时 Text 重建。`WorldMapScene` 的 `renderMapL3`/`renderOverlay` 用单个 Graphics + `g.clear()`、无 Text，不在此列。
  - **回归测试**：`client/test/ui/textTeardown.ui.ts`（headless PIXI，`npm run test:ui`）—— 锁定顶层/嵌套 Text 释放、共享 bake 纹理保留、裸 `removeChildren()` 确实泄漏、ChatScene 反复重渲不跨代堆积。
- **⚠️ 菜单场景生命周期契约（2026-07-07 事故：切换 UI 偶发 `transform is null` + 整页卡死，只能刷新）**：根因是 PIXI7 `Ticker._tick` 在任一监听器抛异常时**中止 update 循环且不再 `requestAnimationFrame`** → 整个 canvas 永久冻结。三条防线：
  - **`SceneManager.onTick` 用 try/catch 包 `scene.update()`**（每场景只记一次错，避免刷屏），`goto()` 亦 try/catch 包 `prev.destroy()`——一次抛出不再能杀死主 ticker / 阻断切场景。
  - **`InputManager` 分发用快照 + 逐 handler try/catch**：旧 `list.forEach(f=>f())` 里一个 handler 抛异常会连坐其后所有 handler（含当前场景的）→ 表现为「点啥都没反应」。
  - **每个 `Scene.destroy()` 必须 `container.destroy({children:true})`**（此前约 20 个菜单场景只 `unsubs.forEach()`，泄漏每个子对象的 `Ticker.shared` 闭包——boil 标题/建筑/单位特效——跨导航累积拖垮帧率）；**异步重绘（`fetch`/纹理 `once('loaded')` 回调）须以 `destroyed` 标志兜底**（`render()` 首行 `if(this.destroyed)return`，是覆盖所有重绘入口的唯一节流点）。`StatePlayerScene` 另需显式 `boardView/unitView/buildingView.destroy()`（各自注销 `Ticker.shared` 特效 tick）。
  - **回归测试**：`client/test/ui/sceneManager.ui.ts`（假 ticker 步进：update 抛错被吞、destroy 抛错不阻断切场景）+ `client/test/inputManager.test.ts`（一个 handler 抛错不连坐其余）+ `scenes.ui.ts` 的 `exercise()` 现断言 **每个场景 `destroy()` 后 `container.destroyed===true`**——任何将来忘记拆容器的场景会当场挂测。
- **⚠️ 场景淡入淡出切换 + 输入冻结契约（2026-07-11 事故：大厅「Store」进商城→点「Shop」页却跳去「Career」；2026-07-12 收窄：淡出淡入不是所有场景切换都需要，子标签/普通返回不该有动画）**：`SceneManager.goto(scene, opts?)` **默认同帧 instant 切换**（不传 `opts` 或不传 `fade`）——覆盖绝大多数导航（大厅各 tab、返回按钮、二级页面）。仅显式传 `{fade:true}` 才走 cross-fade（纸色调 `0xfaf6ee`、峰值透明度 0.92，非早期版本的纯黑全遮罩；fade-out 90ms → swap → fade-in 180ms，`easeInOutQuad`）。目前只有 4 处调用点传 `{fade:true}`：`app.ts` 的 `showGame`/`showGameNet`/`showWorldMap`（进入对局/进入 SLG），以及 `nav.goLobby({fade:true})` 的调用方——`game.ts`（对局放弃/教程完成跳过）、`result.ts`（净斗放弃/结算页返回或再来一局默认落地大厅）、`world.ts`（SLG 地图退出）。新增「进入/退出对局或 SLG」以外的转场**不要**加 `{fade:true}`。淡入期间**新场景在 `new XxxScene()` 时即已构造并订阅输入**，旧场景要等 fade-out 结束的 swap 才 `destroy()`——故约 270ms 内新旧两套 hit-rect **同时在线**。关键坑：**输入不走 PIXI**（`WebAdapter` 直接把 DOM `pointer*` 喂进 `InputManager`），所以纸色遮罩的 `eventMode='static'` 挡不住点击——淡入期一次点击会落到旧场景仍在线的 hit-rect（大厅底栏 Store 隔壁就是 Career）→ 误导航；**instant 切换完全不冻结输入**，只有显式 `{fade:true}` 才会启用下面这套门控。
  - **修复**：`SceneManager` 持有 `InputManager`（`InputGate` 接口），`startTransition` 时 `suppress(true)`、fade 完全结束（或 instant swap / 容错兜底）时 `suppress(false)`——`InputManager.dispatch` 前置 `if(suppressed)return` 丢弃期间所有指针事件。
  - **首击中止 fade**（避免 270ms 死区）：`suppress` 期间的 pointer-**down** 触发 `onSuppressedInput` 钩子 → `SceneManager.skipTransition()` 立即跳到目标场景并解冻；该 down 被吞不下发，且 `swallowNextUp()` 一次性吃掉配对的 pointer-**up**，使这次「中止点击」不会误点到刚挂载的新场景。
  - **回归测试**：`inputManager.test.ts`（suppress 丢弃三通道 / suppressed-down 触发钩子且不下发 / `swallowNextUp` 一次性 / suppress 期间 arm 的 swallow 跨解冻仍生效）+ `sceneManager.ui.ts`（默认 instant 同帧、`{fade:true}` 延迟 swap、instant 取消进行中的 fade、fade-out 中途 retarget、冻结跨 fade 后解冻、首击中止 + 真 `InputManager` 集成：中止后配对 up 被吞、下一击正常送达）。
- **⚠️ 输入订阅遗漏（永久泄漏，非上条的瞬时竞态）契约（2026-07-13 事故：社交页点返回偶发跳到 Career）**：与上条 2026-07-11 的 fade 竞态是**同一症状、不同根因**——`TitlesScene` 构造时直接 `input.onDown(...)`，**丢弃了返回的取消订阅函数**，`destroy()` 也只 `tearDownChildren(container)`，是全代码库唯一没走 `this.unsubs.push(...)` + `destroy()` 里 `unsubs.forEach(u=>u())` 标准模式的场景。后果：每进一次「称号」页就永久多挂一个绑定 Career 侧边栏 `[Stats|Titles|Achievements]` hit-rect 的 down 监听——`InputManager.dispatch()` 广播给**所有**订阅者而非只当前场景，所以之后任何一次点击只要坐标落在那个失效 rect 里（哪怕当前显示的是别的场景的返回按钮），都会额外触发 `onOpenStats()`，与真正点击的场景导航同帧抢跑，最终谁的 `goto()` 后调用谁赢。**排查手法**：在 `_emitDown`/`_emitUp` 里打印 `downs.length`/`ups.length`（正常应恒为 1），`>1` 直接实锤泄漏，比逐场景 code review 快得多。**修复**：给 `TitlesScene` 补齐标准 `unsubs` 数组 + `destroy()` 清理。**教训**：第 21 条只保证了 `container.destroy({children:true})`，没单独强调「每个 `input.onX()` 的返回值都必须存起来并在 `destroy()` 里调用」——两者是独立契约，容器销毁不会自动带走 `InputManager` 的订阅，新增场景时两条都要抄。**审计（2026-07-13）**：逐一核对全代码库约 28 处 `onDown`/`onMove`/`onUp` 调用点，除 `TitlesScene` 外全部规范，未发现同类泄漏。**回归测试**：`client/test/input-subscription-cleanup.test.ts` 静态扫描 `client/src` 全部源文件——任何 `.onDown(`/`.onMove(`/`.onUp(` 调用若不是 `unsubs.push(...)` 包裹的形式即判失败（用原始泄漏那行代码验证过确实会挂测），比逐场景 review 更快锁死这条契约。
- **⚠️ Career 侧边栏 tab 切换的 back 语义（2026-07-13 事故：Career 页点 Titles 标签再点返回，落到 Career/Stats 标签页而非大厅）**：`Stats/Titles/Achievements` 是同一 Career hub 的对等 tab（`ui/widgets/CareerTabs.ts`），不是导航栈——但 `app/nav/game.ts` 里 `goStats`/`goAchievements`/`goTitles` 互相切 tab 时曾各自硬编码 `goTitles(goStats)`/`goTitles(back)`（未传）等写法，把 back 目标写死成"返回 Career 根（Stats）"，导致从任一非根 tab 返回都要多点一次才能出 hub。**修复**：三者统一改为把调用方传入的 `back` 闭包原样透传给对等 tab（`goStats(back)`/`goTitles(back)`/`goAchievements(back)`），只有独立入口（大厅直接进 Achievements、无参 `goTitles()`）才默认落到 `goLobby`/`goStats`。**回归测试**：`client/test/careerNav-backNavigation.test.ts`（Stats→Titles/Achievements 切 tab 后 back 直达大厅、Achievements→Titles 两跳仍透传同一 origin、无参 `goTitles()` 仍保留退到 Stats 的默认行为）。
- **内存看护（MemoryMonitor）**：`app.ts` 启动时 `new MemoryMonitor().install(app.ticker)`，跨场景常驻。每 5 秒采样 `performance.memory.usedJSHeapSize`，超阈值（默认 400MB，`localStorage 'nw_mem_warn_mb'` 可调）即 `console.warn`（`netLog('mem')`）并 dump 各对象池的空闲对象数 + 粗估占用，30 秒内不重复刷屏；微信侧另接 `wx.onMemoryWarning`（移动端真预算闸门，微信运行时常无 `performance.memory`）立即同样 dump。各池经 `cache/poolRegistry.ts` 登记/注销：4 个 `ObjectPool`（unit.circle/building/hand.slot/fx.meteor）由 `ObjectPool` 构造自注册、`drain()` 注销；3 个手工池（unit.stickman/fx.vfx/projectile）在各拥有者构造登记、`destroy()` 注销。**注意：`usedJSHeapSize` 只含 JS 堆、不含 GPU 纹理显存**——故"堆高但池都小"多为泄漏的场景图/闭包（同上次事故），某池 idle 只涨不回落才是池侧泄漏，两类靠这份 dump 区分。**池化范围**：仅战斗内高频生灭对象（stickman 是关键热路径）池化；HP 条/护卫 sprite、HUD 等常驻或低频对象按需 new/destroy，不入池。
