# 验收 S1-9 — 双真机对局（逐 tick 一致 + 断线重连）

> 创建：2026-06-23。Track 3 L3-4 产出。
> **目的**：两台真机走好友房完整一局，验证锁步逐 tick 一致 + 中途断一台在宽限期内 `conn_resync` 重连续打。headless E2E 已绿，但不证明真机双进程一致性，必须人工实测并归档。
> **被测机制**（代码事实，勿改步骤口径）：
> - 锁步中继：gameserver 数据面 WS（`?ticket=` 握手），永不连库，逐帧节拍中继。
> - 重连补帧：`Room.resume(conn, lastFrame)` → 服务端发 `conn_resync { seed, startFrame, log(frame>lastFrame), curFrame }`；断线方在 `GRACE_MS` 宽限窗内重连即可补齐缺帧续打，双方都回连后恢复节拍器。
> - 一致性判定：双端按相同 `seed` + 相同指令日志确定性推演，逐 tick 的状态 hash 必须一致（无 mismatch）。

## 1. 前置条件

- [ ] 服务端全栈可用（metaserver / gateway / matchsvc / gameserver）—— 当前 main 构建或 Track 1/2 合入后构建。
- [ ] 两台真机（建议跨平台：iOS + Android，或一台 + 一台 Web），各装/开同一版本客户端。
- [ ] 两个测试账号（A、B），已互为好友（社交系统）。
- [ ] 客户端开启逐帧 hash 校验/日志（若有 debug 开关）；或准备抓 WS 帧日志。
- [ ] 网络可控：能对其中一台手动断网（飞行模式/拔 WiFi）再恢复，用于重连测试。

---

## 2. 测试用例

### TC-1 完整一局逐 tick 一致（基线）
1. 设备 A 登录账号 A，创建好友房，邀请 B。
2. 设备 B 登录账号 B，加入房间。
3. 双方就绪开局；正常对战至分出胜负（settled）。
4. 全程不断网。

**期望**：
- 双端开局收到相同 `seed` 与 `startFrame`。
- 对局全程双端逐 tick 状态 hash 一致，无 mismatch 告警。
- 双端同时、一致地结算（胜负、奖励、段位变化一致）。
- match-report 上报成功，meta 侧结算一致。

### TC-2 中途断一台 + 宽限期内重连续打
1. 重复 TC-1 步骤 1–3，进入对局中段（如第 N 帧，建议对局进行 20–40s 后）。
2. 对设备 B 手动断网（飞行模式），保持 `< GRACE_MS`（默认宽限窗内，建议 10–30s）。
3. 期间观察设备 A：对局应暂停节拍（等待对方）或按设计表现，不应崩溃/误判胜负。
4. 设备 B 恢复网络并重连同一房间。

**期望**：
- B 重连后收到 `conn_resync`，`log` 含断线期间缺失帧（`frame > lastFrame`），`curFrame` 与 A 对齐。
- B 补齐缺帧后状态 hash 与 A 一致，续打无错位。
- 双方都回连后节拍器恢复，对局正常打完并一致结算。

### TC-3 超时未重连（边界，可选）
1. 同 TC-2，但设备 B 断网时长 **超过** `GRACE_MS`。

**期望**：按设计判定（如判 B 掉线负/房间结束），双端表现一致，无悬挂房间；记录实际行为。
