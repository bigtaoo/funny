# 验收 C6-c — 全平台真机冒烟 + 防刷复核

> 创建：2026-06-23。Track 3 L3-6 产出。
> **目的**：四平台（iOS / Android / 微信 / CrazyGames）真机走核心路径无阻断；防刷 e2e 复核——本地改 `wallet.coins` 后任何花币动作被 commercial `$gte` 守卫拒绝。
> **被测机制**（代码事实，见 `server/commercial/src/service.ts`）：
> - 扣币走单文档原子守卫 `findOneAndUpdate({_id, coins: {$gte: cost}}, {$inc:{coins:-cost}})`；余额不足守卫失败 → 返回 `INSUFFICIENT_FUNDS`，**不扣不发货**。
> - 消费 `orderId` 幂等、充值 `receiptId` 幂等。
> - 钱包余额是**服务器权威**（commercial 持有），客户端显示值改不动服务端真值（呼应 S0-8 权威段）。

## 2. 防刷复核（本地改币花不出去）

### TC-防刷-1 改本地余额后花币被拒
1. 任一端登录，记录服务端真实 `coins`（如 100）。
2. 用调试/抓改手段把**客户端显示**余额改大（如 999999）。
3. 触发一个**花币动作**且其 cost > 服务端真实余额（如买一个 cost=500 的道具）。

**期望**：
- 服务端 commercial `$gte` 守卫失败 → 返回 `INSUFFICIENT_FUNDS`。
- **不扣币、不发货**；客户端显示被服务端真值纠正/拒绝。
- 服务端 `coins` 真值不变（仍为 100）。

### TC-防刷-2 幂等复核（可选）
1. 同一 `orderId` 的消费请求重发两次。

**期望**：第二次幂等返回首次结果，不重复扣币/发货。

### TC-防刷-3 存档 push 不可改钱包（呼应 S0-8 TC-3）
1. 构造存档 push body 塞入伪造 `wallet`。

**期望**：被同步段白名单丢弃，钱包真值不变。
> 已由 `save.e2e.test.ts` / `commercial/test/service.e2e.test.ts` 覆盖；真机抽查一次。
