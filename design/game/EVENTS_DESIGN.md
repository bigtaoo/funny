# Notebook Wars — 活动 / Live-ops 系统设计

> 状态：设计中 · 权威：本文（运营活动**编排/配置/生命周期**的单一入口）· 更新：2026-06-21
>
> **权威边界**：本文拥有**活动的调度、配置模型、生命周期、客户端呈现**。**不**另造子系统——发奖复用 [`OPS_DESIGN.md`](OPS_DESIGN.md) 邮件路径、任务计数复用 [`RETENTION_DESIGN.md`](RETENTION_DESIGN.md)/[`ACHIEVEMENT_DESIGN.md`](ACHIEVEMENT_DESIGN.md) 的 statKey 累加、限定直购复用 [`COMMERCIAL_DESIGN.md`](COMMERCIAL_DESIGN.md) 商店、数字落 [`ECONOMY_NUMBERS.md`](ECONOMY_NUMBERS.md)、金币红线归 [`ECONOMY_BALANCE.md`](ECONOMY_BALANCE.md)（ADR-014）。**活动是叠在既有系统上的容器,不是平行的第二条路径。**

---

## 1. 定位

活动 = **有时效的运营内容容器**,用配置点亮一段时间内的额外目标/奖励/售卖,制造"现在不玩就错过"的回访动机,并给经济/留存一个**可运营的调节旋钮**(双倍掉落、限定皮肤、节日主题)。

它回答 ECONOMY_BALANCE §299 的待决问:「是否引入限时活动池 / 双倍掉落」——**引入,但严格受控**(§6)。

**基调**:与全局克制风格一致——活动**叠加正向收益,从不削弱常态**(不搞"活动期外惩罚");可错过不可剥夺。

---

## 2. 活动类型（首批）

| 类型 | 玩法 | 奖励载体 | 经济影响 | 优先级 |
|---|---|---|---|---|
| **登录活动** | 活动期内累计登录 N 天领里程碑 | 软通货为主(体力/材料/碎片) | 受留存预算约束 | P0 |
| **活动任务** | 活动期专属任务清单(打 X 局/通 Y 关/围攻 Z 次) | 软通货 + 活动积分 | 计数复用 statKey | P0 |
| **双倍 / 加成期** | 限时 PvE 掉落 ×N / 体力消耗 ×0.5 等乘子 | 不直接发奖,改既有产出乘子 | **最敏感,严格封顶**(§6) | P1 |
| **限定直购** | 活动期商店上架限定皮肤/礼包,过期下架 | commercial 商店 IAP/金币 | 变现,不破皮肤稀有度铁律 | P1 |
| **积分兑换商店** | 活动积分(活动内产)兑限定奖励 | 活动积分 → 限定物 | 积分活动期清零,不入持久经济 | P2 |
| **节日主题** | 美术/文案换皮 + 轻奖励 | 软通货 | 极小 | P2 |

> 积分(`eventPoints`)是**活动局部货币**,活动结束清零,**不进 SaveData 持久经济**,故不参与金币通胀推演。

## 3. 配置模型（数据驱动,admin 下发）

活动是**配置对象**,不硬编码进客户端。配置由 admin 后台编辑下发(复用 OPS 的运营能力),服务器权威解释。

```
interface EventDef {
  eventId: string;            // 稳定唯一 id
  type: EventType;            // 见 §2
  window: { startTs; endTs };  // 活动时间窗(服务器时钟权威)
  i18n: { titleKey; descKey };// 文案走 i18n,不内嵌明文
  theme?: string;             // 节日换皮资源 id(可选)
  tasks?: EventTask[];        // 活动任务(statKey + 阈值 + 奖励)
  milestones?: Milestone[];   // 登录/积分里程碑
  multipliers?: { dropRate?; staminaCost?; ... };  // 加成期乘子(受 §6 封顶)
  shopItems?: string[];       // 限定直购 itemId(commercial 商店,带同一 window)
  rewards: RewardSpec[];      // 统一奖励规格(发奖走邮件,§5)
}
```

- **单一来源**:可调数字(乘子上限、积分产出、里程碑阈值)集中 [`ECONOMY_NUMBERS.md §14`](ECONOMY_NUMBERS.md)(待建节),配置只引用不另定。
- **校验**:服务器加载配置时校验 window 合法、乘子不超封顶、rewards 在白名单内——非法配置拒绝上线(防运营误操作放出超量奖励)。

## 4. 生命周期（服务器权威时钟）

```
draft(admin编辑) → scheduled(已排期未到点) → active(窗口内) → ended(过点停止发放) → settled(结算/清积分/归档)
```

- **时钟权威在服务器**(同 RETENTION dayKey / S1 节拍器思路),客户端只读展示倒计时,**所有发放/计数/兑换按服务器时间二次校验**——客户端改本地时间无效。
- **active→ended**:窗口一过立即停止新发放;**已获得的奖励/积分进入领取宽限**(如结束后 7 天仍可领邮件,过期回收)。
- **settled**:活动积分清零、限定商品下架、活动数据归档(供 analytics 复盘)。
- 与**赛季时钟解耦**:活动有自己的 window,**不**和天梯6周/SLG大区2月赛季时钟绑定(SEASON_OVERVIEW 的"两条独立时钟"再加活动这条第三时钟,互不触发)。活动可跨赛季、可赛季内,纯由 window 决定。

---

## 5. 奖励发放（复用邮件,不造第二条路径）

> 铁律(同 OPS5):**活动奖励一律走邮件**,服务器据活动结算创建系统邮件,玩家领取时才经 commercial/inventory 入账(幂等)。**不新建发奖通道。**

- 个人里程碑/任务奖励:达成即服务器发系统邮件给该玩家。
- 全服活动奖励:复用 OPS 全服邮件 fan-out(分批),与补偿全服奖励同一基础设施。
- 限定直购:走 commercial 正常购买流(扣币/IAP → 发货幂等),活动只控上下架窗口。
- 领取幂等键 = `eventId + milestoneId + accountId`,与 OPS 邮件领取幂等同构。

---

## 6. 经济约束（ADR-014,反通胀红线）

> 活动是最容易冲垮 F2P ~300 金币/月预算的口子,锁死规则:

1. **不新增金币龙头**:活动金币产出(若有)计入 ECONOMY_BALANCE §1 月度预算,不另开持续 faucet——与 ADR-011(留存)同口径。活动主发**软通货 + 限定皮肤碎片 + 活动积分**。
2. **双倍/加成期有硬封顶**:掉落乘子、加成时长、同时生效活动数都设上限(数字 ECONOMY_NUMBERS §14),且只作用于**受体力闸门约束的 PvE 产出**——体力总闸在,双倍也翻不出天(ADR-009 体力为总闸门)。
3. **限定直购不破皮肤稀有度铁律**:高级 epic/legendary 仍**不**因活动变成金币直购(ECONOMY §皮肤政策);活动限定是**新增 SKU / 折扣**,不是降级稀有度。
4. **积分活动期清零**:活动局部货币不沉淀进持久经济。
5. **PvP 硬墙恒不读活动加成**:活动只注入 PvE/SLG/账号资源,公平 PvP 绝不受影响(ADR-009 红线)。

---

## 7. 防刷（复用既有,不开客户端写口）

- **任务计数**:活动任务进度只在**服务器权威结算点**累加(PvE 结算 / `match/report` / 围攻结算),复用 RETENTION R6 / ACHIEVEMENT A2 的 statKey 累加链,**绝不信客户端自报**。
- **领取/兑换**:服务器按 window + 幂等键二次校验,重复领取无效。
- **加成生效**:乘子由服务器在产出结算时套用,客户端只展示。

---

## 8. 客户端呈现

- **活动入口**:大厅活动 banner/红点(受 ONBOARDING 渐进解锁约束——新手期不弹活动,避免信息过载)。
- **活动面板**:列当前 active 活动、倒计时、任务进度(服务器下发)、里程碑领取、积分商店。
- **限定商店**:活动 SKU 并入 ShopScene,带"限时"标与倒计时。
- **i18n**:`event.*` 全语种(zh/en/de);活动文案走 i18n key,**不**在配置里塞明文,便于多语言运营。
- **离线/无网**:活动只读展示,领取走 API;无服务端配置时活动入口隐藏(同其他联网功能降级)。

---

## 9. 埋点（字段权威归 ANALYTICS）

活动曝光 → 入口点击 → 任务完成 → 里程碑领取 → 限定购买 → 活动留存(活动期 vs 平时 DAU)。**事件字段定义归 [`ANALYTICS_DESIGN.md`](ANALYTICS_DESIGN.md)**,本文只列该埋哪些节点;用于活动效果复盘(settled 后分析)。
